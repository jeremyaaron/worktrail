Project management reference app

Worktrail

Worktrail is a focused project management app for daily execution, project operation, bounded work breakdown, workspace portfolio review, cycle planning and closeout, collaboration, dependencies, files in context, and published status snapshots.

It is also a reference implementation for an Angular static frontend, a TypeScript API boundary, domain services, Postgres persistence, and production-shaped local operations.

The product

A working operating surface for team project work.

Worktrail is intentionally compact, but its workflows are real. The app gives a team a daily starting point, durable ways to find and shape work, project-level operating views, and published status artifacts that preserve what was true when a report was shared.

Start the day

Open the work and updates that need attention.

  • My Work summarizes assigned work, due dates, dependency-blocked work, and stale items.
  • Action Inbox collects mentions, assignments, watcher updates, and dependency changes.
  • Local actor switching exercises owner, maintainer, and contributor paths.
  • Notifications link back to the relevant work item or project context.
Review the portfolio

Compare active projects before diving into details.

  • Portfolio summarizes active project health, overdue work, dependency pressure, and report freshness.
  • Attention sections group delivery risk, communication freshness, current execution, and dependency pressure.
  • Project rows link into Work, Planning, Reports, milestone review, cycle review, and latest report detail.
  • Seed data demonstrates blocked projects, a healthy comparison project, fresh reports, and missing reports.
Find and shape work

Use applied filters as a product contract.

  • Workspace and project work lists use URL-backed filters, active chips, exact counts, and server-backed pages.
  • Saved views support personal, workspace-shared, and project-scoped operating lenses.
  • Pinned shortcuts keep the most important saved views visible.
  • Filtered CSV exports include all matching rows under a 10,000-row synchronous limit, independent of the visible page.
Break down outcomes

Turn a deliverable into independently operated child work.

  • Top-level work can contain one direct child level inside the same project.
  • Parent summaries show direct child state and estimate context without changing the parent's own fields.
  • Parent and child context stays visible on detail, lists, board cards, My Work, reviews, reports, and exports.
  • Containment remains separate from blockers, workflow status, cycles, milestones, and dependencies.
Operate projects

Keep project decisions close to delivery state.

  • Project shell navigation keeps Overview, Work, Board, Planning, Reports, and Settings in context.
  • Angular CDK board interactions preserve durable server-validated order.
  • Owner and maintainer bulk triage applies scoped project updates with per-row results.
  • Relationships surface blocked-by, blocking, and related work across the workspace.
Keep files in context

Attach bounded evidence directly to the work.

  • Active members can upload supported files with per-file progress and retry.
  • Compact metadata shows filename, type, size, uploader, time, and remaining capacity.
  • Authorized downloads are integrity-checked and always forced as file attachments.
  • Uploader and role-aware removal retains readable activity; archived projects stay download-only.
Share status

Separate live planning from published evidence.

  • Planning, Milestone Review, and Cycle Review pages are labeled as live views of current state.
  • Cycles track methodology-neutral execution windows with scoped work and target points.
  • Guided closeout previews mixed scope, carries unfinished work forward atomically, and preserves an immutable cycle result.
  • Report drafts separate generated evidence from editable narrative.
  • Published reports preserve immutable milestone, cycle, risk, and recent-work snapshots while linking to current work.
  • Markdown copy, Markdown download, and print-friendly report detail pages support sharing.

Architecture

Local-first now. Cloud-shaped where it matters.

Worktrail runs locally with Angular, Express, and Postgres. The important reference value is in the boundaries: frontend state is explicit, HTTP handlers are adapter-ready, domain rules live in services, persistence is relational, and operations have concrete build and readiness paths.

FrontendAngular standalone route components, lazy route chunks, static build output, URL-backed query state, and responsive work surfaces suitable for S3/CloudFront hosting.
API boundaryShared DTO contracts, explicit request validation, bounded binary and JSON responses, transport-neutral endpoint handlers, Express local adapter, and OpenAPI documentation.
DomainAttachment policy and authorization, project planning, bounded work hierarchy, cycle review, delivery health, relationships, notifications, saved views, CSV flows, report snapshots, and service-level permissions.
PersistencePostgres metadata plus immutable bytes behind an object-store port, with Drizzle migrations, trigram search indexes, repeatable-read pages, deterministic seeds, lifecycle snapshots, activity, notifications, relationships, and saved views.
OperationsProduction preview, runtime config validation, liveness/readiness endpoints, CI lint/typecheck/test/build, and local smoke-test guidance.

Operations

Built artifacts, readiness signals, and documented run paths.

The production preview compiles the API and Angular app, serves built frontend assets from Express, keeps `/api/*` separate from SPA fallback, requires an explicit production database URL and absolute attachment storage path, and exposes separate process liveness and database readiness checks.

Preview

One command from source to built runtime.

npm run preview builds contracts, API, and web artifacts, then starts production mode.

Readiness

Separate process and database checks.

/api/health/live confirms the API process; /api/health/ready checks Postgres.

API docs

A checked-in OpenAPI reference.

Implemented routes, local actor headers, representative DTOs, readiness behavior, and errors are documented.

Runbook

Operational commands with explicit caveats.

Setup, migrations, reset safety, preview, deep-link checks, and cloud mapping are covered.

Current scope

Useful reference app, not hosted SaaS.

Worktrail is candid about what it is not. The value today is a working product surface and production-shaped local architecture, not an internet-safe deployment.

No production auth yet Authentication is represented by local actor headers and the top-bar actor selector. Real identity, invitations, tenant provisioning, and custom roles are deferred.
No hosted infrastructure yet The production preview is for local operational inspection. Managed deployment, S3, shared multi-instance file storage, secrets, observability, background workers, and internet-safe hosting are not included. Interactive work lists are paged, but exact counts and deep offsets have a cost; complete boards, aggregate reads, and synchronous exports retain documented limits.
No file-content security layer yet Attachments are bounded and signature-checked, but they are not malware-scanned, previewed, rendered inline, indexed, versioned, retained by policy, or transferred through background and resumable jobs. Production authentication is also deferred.

Run locally

Clone, migrate, seed, and start.

The default local path uses Postgres, deterministic seed data, the Angular dev server, and the local Express API. 

npm install
npm run db:start
npm run db:migrate
npm run db:seed
npm run dev

# production preview
export DATABASE_URL=postgres://worktrail:worktrail@localhost:5432/worktrail
export WORKTRAIL_ATTACHMENT_STORAGE_PATH=/absolute/path/to/worktrail-attachments
npm run preview
